# Dokku Dashboard

A small, self-hosted dashboard for checking every application on one Dokku
server. It presents a compact responsive grid with application state, uptime,
current and seven-day peak CPU/RAM, 24-hour charts, container details, root
filesystem size, Dokku build-cache size, and host RAM/disk distribution.
Application names link directly to their configured domains. The page refreshes
automatically every 30 seconds and is protected by one shared password.

The production image uses a digest-pinned Node.js 22.23.1 Alpine base, runs as
UID/GID 32767, and is approximately 58 MB. The repository does not currently
ship a screenshot asset; the interface is available locally through the mock
preview described below.

## How it works

- Express serves the login, dashboard, and JSON APIs; the frontend is plain
  HTML, CSS, and JavaScript.
- The Docker socket supplies Dokku-labelled containers, live container stats,
  image/root-filesystem sizes, and `cache-<app>` volume usage.
- Linux `/proc/meminfo`, Node system APIs, and filesystem stats supply host
  capacity data.
- SQLite at `/app/data/metrics.sqlite` retains detailed 24-hour samples and
  hourly seven-day peaks. The file survives deployments through a host mount.
- Express sessions provide password authentication. Sessions are in memory, so
  a restart signs users out without affecting metrics.

## Prerequisites

- A Linux server with Docker and Dokku 0.34 or newer.
- Administrative shell access to the Dokku host.
- A Git client locally and SSH access to Dokku for deployment.
- Optional: a DNS hostname and the `dokku-letsencrypt` plugin. A trusted HTTPS
  endpoint is strongly recommended and is required for production secure login
  cookies to work reliably.

## Install on Dokku

Clone the repository on your workstation:

```sh
git clone https://github.com/mrmakko/dokku-dash.git
cd dokku-dash
```

Run the following on the Dokku host. The explicit port mapping is important:
Dockerfile `EXPOSE 5000` describes the container port, while public HTTP and
HTTPS must continue to use ports 80 and 443.

```sh
dokku apps:create dashboard
dokku builder:set dashboard selected dockerfile
dokku ports:set dashboard http:80:5000 https:443:5000

sudo install -d -o 32767 -g 32767 -m 0770 \
  /var/lib/dokku/data/storage/dashboard-metrics
dokku storage:mount dashboard \
  /var/lib/dokku/data/storage/dashboard-metrics:/app/data

SOCKET_GID=$(stat -c '%g' /var/run/docker.sock)
dokku docker-options:add dashboard deploy,run \
  "--volume /var/run/docker.sock:/var/run/docker.sock"
dokku docker-options:add dashboard deploy,run "--group-add $SOCKET_GID"
```

The image and persistent directory deliberately use UID/GID 32767. The Docker
socket group is host-specific, so calculate it rather than copying a number
from another server.

Set required configuration from a private administrative session. This pattern
keeps literal secrets out of shell history; as with most environment CLIs,
values can still be briefly visible to privileged process inspection while the
command runs.

```bash
set +o history
read -r -s -p 'Dashboard password: ' DASHBOARD_PASSWORD; printf '\n'
read -r -s -p 'Long random session secret: ' SESSION_SECRET; printf '\n'
dokku config:set --no-restart dashboard \
  NODE_ENV=production \
  METRICS_DB_PATH=/app/data/metrics.sqlite \
  DASHBOARD_PASSWORD="$DASHBOARD_PASSWORD" \
  SESSION_SECRET="$SESSION_SECRET"
unset DASHBOARD_PASSWORD SESSION_SECRET
set -o history
```

Use a long random `SESSION_SECRET` generated by a trusted password manager or
cryptographic generator. `DASHBOARD_PASSWORD` is the shared login password.

Add the Dokku remote and deploy from the cloned repository:

```sh
git remote add dokku dokku@YOUR_DOKKU_HOST:dashboard
git push dokku HEAD:master
```

### Domain and TLS

If you have a hostname, point its DNS record at the server, then configure it:

```sh
dokku domains:set dashboard dashboard.example.com
```

If `dokku-letsencrypt` is already installed, configure its contact address and
enable a certificate after DNS resolves correctly:

```sh
dokku letsencrypt:set dashboard email admin@example.com
dokku letsencrypt:enable dashboard
```

Otherwise use the server's existing TLS workflow. Do not expose the login over
unencrypted public HTTP.

## Verify the installation

```sh
dokku ps:report dashboard
dokku ports:report dashboard
dokku storage:report dashboard
dokku docker-options:report dashboard
dokku run dashboard id
dokku run dashboard sh -c 'test -w /app/data'
dokku logs dashboard --tail
```

The runtime identity should be UID/GID 32767, the write probe should succeed,
and the logs should show the server listening on the assigned `$PORT`. Open the
configured HTTPS hostname, sign in, and confirm that application cards and
host usage appear. Metrics charts populate as collection samples accumulate.

## Local development

Install dependencies and run all tests:

```sh
npm ci
npm test
```

Launch the deterministic mock dashboard without Docker, Dokku, authentication,
or a local metrics database:

```sh
npm run preview
```

Open <http://127.0.0.1:4173>. Set `PREVIEW_PORT` to use another local port. The
preview server and mock data are excluded from the production image.

## Upgrades

Review changes, test, and push the new revision through the same Dokku remote:

```sh
git pull --ff-only
npm ci
npm test
git push dokku HEAD:master
```

Do not recreate or remove `/var/lib/dokku/data/storage/dashboard-metrics`
during an upgrade. Recalculate the Docker socket GID when migrating to another
host. The pinned image digest makes the Node base reproducible; updating it
should be an intentional, tested repository change.

## Metrics backup and restore

`/app/data` contains the SQLite database plus possible WAL sidecars. For a
simple consistent backup, briefly stop writes and archive the entire storage
directory:

```sh
dokku ps:stop dashboard
sudo tar -C /var/lib/dokku/data/storage/dashboard-metrics \
  -czf dashboard-metrics-$(date +%F).tar.gz .
dokku ps:start dashboard
```

Store backups away from the Dokku host and test restoration. Restore while the
application is stopped, then restore ownership to `32767:32767` before starting
it. Never copy only the main SQLite file while the application is writing; that
can omit committed WAL data.

## Security

Mounting `/var/run/docker.sock` gives this application highly privileged
control over the Docker host, even though its process is non-root. Run only
reviewed code, keep the dashboard behind HTTPS and network controls where
possible, and protect the shared password carefully. This project is intended
for a small trusted environment; it does not provide individual accounts,
roles, audit trails, brute-force protection, or a durable session store.

## Troubleshooting

- **`SQLITE_READONLY` or database open errors:** confirm the storage mount is
  present and owned by `32767:32767`; rerun the `/app/data` write probe.
- **No applications or container metrics:** confirm the socket mount and that
  `--group-add` matches `stat -c '%g' /var/run/docker.sock`, then redeploy.
- **Public port 5000 or an nginx 502:** rerun the explicit `ports:set` command,
  inspect `dokku ports:report`, and check application logs.
- **Login returns to the login page:** use the configured HTTPS hostname;
  production cookies are secure and are not retained over plain HTTP.
- **`npm: not found` inside the container:** expected. Package managers are
  removed from the final image; Dokku and the Dockerfile launch Node directly.
- **Charts are initially empty:** wait for metric collection. Existing history
  is preserved only when `/app/data` is mounted correctly.

For production hardening, named-storage migration on newer Dokku releases,
backup validation, domain rollback, and deeper recovery steps, see
[DEPLOYMENT.md](DEPLOYMENT.md).
